# 比赛收尾作战计划

这份文档用来保证后续即使没有人实时带着做，项目也能继续推进到可演示、可答辩、可交付。

## 1. 当前默认策略

| 事项 | 默认选择 |
| --- | --- |
| 第一轮联调 | 本机或同一局域网联调 |
| AI 回复 | 后端继续使用 `TREEHOLE_AI_PROVIDER=mock` |
| 公网部署 | 暂时后置，等前端和 AI 主流程跑通后再做 |
| 前端代码 | 由前端同学单独维护 |
| 后端仓库职责 | 提供接口、数据库、Swagger、协作文档和后端测试 |

最重要的原则：先跑通一条完整演示链路，再补细节。

## 2. 剩余任务优先级

### P0：必须先完成

| 任务 | 负责人 | 完成标准 |
| --- | --- | --- |
| 后端主流程可启动 | 后端 | `.\scripts\run-test.ps1` 成功，`http://localhost:8080/swagger-ui.html` 可打开 |
| 前端拿到接口文档 | 后端/前端 | 前端同学看过 `前端对接.md` 和 `接口契约文档.md` |
| 用户主流程跑通 | 后端/前端 | 匿名身份、创建会话、发送文本消息、查询消息、星空图数据全部能调通 |
| 管理员登录跑通 | 后端/前端 | `admin / admin123456` 能登录并拿到 token |
| AI 不阻塞联调 | 后端/AI | 后端 mock 回复可用，真实 AI 未完成时前端仍可开发 |

### P1：重要，影响演示完整度

| 任务 | 负责人 | 完成标准 |

### P2：收尾增强

## 3. 三人分工

### 后端

- 保证后端能启动、测试通过、Swagger 可用。
- 保证接口字段和 `docs/接口契约文档.md` 一致。
- 联调时提供 `http://<后端电脑局域网IP>:8080`。
- 公网部署前负责改默认管理员密码、设置 `TREEHOLE_JWT_SECRET` 和数据库密码。

### 前端

- 按 `docs/前端对接.md` 做接口封装。
- 把后端地址统一放到 `API_BASE_URL`，不要散落写死。
- 用户 token 和管理员 token 分开存。
- 遇到接口问题时提供请求地址、请求体、响应体和浏览器 Network 截图。

### AI

- 第一轮不阻塞前端，先按 `docs/AI-RAG对接说明.md` 准备固定 JSON 返回。
- 不直接查后端数据库。
- 准备服务地址、鉴权方式、超时时间、request/response 示例。
- 真实 RAG 做好前，可以先交一个 HTTP mock 服务。

## 4. 每日同步格式

每天晚上用下面格式同步，3 到 5 分钟即可：

```text
我昨天做了：
我今天要做：
我现在卡在：
我需要谁配合：
```

不要只发“我改好了”。每次交接都要说清楚别人下一步怎么接。

## 5. 最小演示脚本

```text
1. 打开前端页面
2. 创建匿名身份
3. 创建一个聊天会话
4. 输入“我最近因为考试很焦虑”
5. 展示 AI 回复和转念建议
6. 打开情感星空图，看见 ANXIOUS 点位
7. 登录管理员后台
8. 查看风险事件、规则和心理援助资源
```

这条链路能稳定跑通，就已经具备第一版答辩基础。

## 6. 公网部署前检查

公网部署前必须确认：

- 不使用默认管理员密码。
- `TREEHOLE_JWT_SECRET` 改成生产用随机长字符串。
- 数据库密码、AI API Key 不写进 GitHub。
- `TREEHOLE_PUBLIC_BASE_URL` 改成真实公网后端地址。
- 上传目录 `TREEHOLE_DOODLE_DIR` 有读写权限。
- Swagger 是否对公网开放由团队决定。
