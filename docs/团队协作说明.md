# 数字树洞团队协作说明

这份文档是给你们 3 个人一起用的：后端、前端、AI。

目标只有一个：让大家知道“现在该做什么、改完怎么交接、出现问题怎么说”，避免项目推进时互相等、互相猜。

## 1. 你们三个人各自负责什么

### 后端

- 提供接口
- 提供数据库表和演示数据
- 提供 Swagger 和接口文档
- 和前端联调
- 和 AI 同学联调真实服务

### 前端

- 做用户页面
- 做管理员页面
- 做涂鸦画板
- 做情感星空图渲染
- 按接口文档发请求

### AI

- 提供真实 AI HTTP 服务
- 让返回字段和后端约定一致
- 给出超时、失败、降级说明
- 如果要写 RAG，先看 `docs/AI-RAG对接说明.md`

## 2. 现在最推荐的协作节奏

### 第一步：所有人都看同一份文档

- 后端看：`README.md`
- 前端看：`docs/前端对接.md`
- 你自己学项目结构看：`docs/小白上手指南.md`

### 第二步：每个人的输出不要只发一句“我改好了”

发消息时尽量带这 3 个信息：

- 我改了什么
- 你现在要怎么接
- 还有什么没做完

一个最简单的例子：

```text
我这边把匿名会话接口和聊天会话接口做好了。
前端现在可以先接 POST /api/v1/visitors/session 和 POST /api/v1/chat/sessions。
还没接真实 AI，现在返回的是 mock/fallback。
```

## 3. GitHub 最小协作规则

这个项目刚初始化成 Git 仓库。你们后面把它推到 GitHub 后，直接按下面规则来就行。

### 分支怎么起名

统一用这种格式：

```text
feat/backend-chat
feat/frontend-home
feat/ai-http-adapter
fix/chat-message-bug
docs/contest-materials
```

### 提交信息怎么写

不用写复杂英文，先写清楚就行：

```text
feat: 完成匿名会话接口
feat: 新增聊天消息分页查询
fix: 修复涂鸦上传大小校验
docs: 补充前端对接说明
```

### 每次提交前至少做 2 件事

- 能启动就启动一次
- 改了接口就自己点一次 Swagger

### 什么时候提 PR

满足下面任一条就可以提：

- 一个小功能做完了
- 想让队友帮你看看代码
- 你改动比较多，怕后面冲突

PR 模板已经放在：

```text
.github/pull_request_template.md
```

## 4. 前后端联调时最容易卡住的点

### `localhost` 用错

如果前端不在你电脑上，`localhost` 不能用。

这时候后端地址要改成你电脑的局域网 IP，例如：

```text
http://<后端电脑局域网IP>:8080
```

### token 没带

除了匿名会话接口和管理员登录，其他接口基本都要带：

```http
Authorization: Bearer <token>
```

### 字段名自己猜

不要靠猜。优先看：

- `docs/接口契约文档.md`
- Swagger 页面

## 5. 后端和 AI 同学怎么约定

后端需要的是一个稳定的 HTTP 服务，不是“差不多能跑”的口头描述。

最少要先说清楚这几个：

- 请求地址是什么
- 请求体字段是什么
- 返回体字段是什么
- 超时多久算失败
- 失败时返回什么

AI 同学如果要写 RAG，不需要直接查后端数据库。后端会在每次调用 AI 时发送当前输入和最近 6 条上下文。具体字段看：

```text
docs/AI-RAG对接说明.md
```

如果真实 AI 还不稳定，当前项目允许继续使用 mock/fallback，不阻塞前端联调。

## 6. 每晚最好同步一次

不用开很正式的会，3 到 5 分钟就够。

每个人各说 3 句：

- 我今天做了什么
- 我明天要做什么
- 我现在卡在哪里

## 7. 当前最适合你做的事

如果你是这个项目的后端初学者，先不要想“我要不要重构”。

先做这几个最实在：

1. 把项目跑起来
2. 打开 Swagger
3. 跑一次匿名会话接口
4. 跑一次发送消息接口
5. 看懂 `Controller -> Service -> 数据库` 这条线
